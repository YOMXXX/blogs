## 1. Cursor Composer 2：编程模型进入专用化时代

**核心事件**：Cursor 发布技术报告（arXiv 2603.24477），详细披露 Composer 2 的训练过程——基于 Kimi K2.5 Mixture-of-Experts 架构专项微调，在 SWE-bench Multilingual 达到 73.7%、Terminal-Bench 达到 61.7%。

**为什么重要**：这是 AI 编程工具从"调用前沿 API"转向"自研领域专用模型"的标志性事件。Cursor 面临的压力是真实的：Claude Opus 4.6 的 per-token 成本让重度 agentic 用户的月均 API 支出很快超过订阅费。Composer 2 以捆绑订阅的形式解决了这个问题，同时在 agentic 代码任务上的性能与旗舰通用模型持平。

**给开发者的启示**：如果你是 Cursor Pro 用户且重度使用 Composer 功能，切换到 Composer 2 是一个无成本的显著升级。在 Cursor 设置 → Models 中即可切换。非 Cursor 用户则要注意这一趋势：IDE 厂商开始构建自己的模型护城河，未来"最好的编程 AI"可能不再是通用大模型排行榜上的第一名。

**延伸阅读**：[Cursor Composer 2 技术报告](https://arxiv.org/abs/2603.24477)，本周本站 workshop 专文：「Cursor Composer 2 实战：用 Kimi K2.5 MoE 驱动的 Agentic 编程模型」

## 2. Gemini 3.5 Pro 延期至 7 月：agent 能力成为新的发布门槛

**核心事件**：Google 将 Gemini 3.5 Pro 从 6 月推迟到 7 月，原因是内测中 agent 任务表现未达预期，需要更多时间调整。

**数字背后**：Gemini 3.5 Flash（5 月已发布）在 WebArena 上约 58%，而 Claude Opus 4.8 约 68%、GPT-5.5 约 65%。对 Google 旗舰模型 Pro 版本而言，这个差距不可接受。延期是为了把短板补上再面对市场。

**更大的含义**：这标志着大模型竞争的判断标准已经迁移——2023 年看 MMLU，2024 年看 MMLU+代码，2026 年看多步骤 agent 任务成功率。通用基准不再是发布的充分条件，必须在 agent 场景下"能用"才算合格。

**7 月值得关注的问题**：
- Gemini 3.5 Pro 的 SWE-bench 和 WebArena 成绩能否追平 Claude Opus 4.8？
- 2M token 长上下文在 agent 场景下的实际表现（vs. Claude 的 200k）
- 推理 API 定价是否有竞争力

## 3. Qwen-AgentWorld：用 LLM 「想象」环境来训练 Agent

**论文**：Qwen-AgentWorld: Language World Models for General Agents（arXiv 2606.24597，阿里 Qwen 团队）

**核心思路**：与其在真实环境中收集训练数据（慢、贵、不可重置），不如训练一个 LLM 来**预测环境如何响应 agent 动作**，再用这个"语言世界模型"（LWM）来训练 agent。

**最有趣的发现**：经过 LWM warm-up 的模型，即使不做任何 agent 任务专项微调，在 agentic 基准上的零样本性能就有显著提升（WebArena +6.8%）。这说明"理解环境如何响应动作"本身就是有效的 agent 能力预热。

**实际意义**：
- 35B-A3B MoE 版本已在 HuggingFace 开源，可本地部署
- AgentWorldBench 涵盖 7 种数字环境，是评估通用 agent 能力的新基准
- 为后续的 agent 训练研究提供了高质量的语言世界模型基础

## 4. 12 个 Agent 记忆系统系统性评估：没有赢家

**论文**：Are We Ready For An Agent-Native Memory System?（arXiv 2606.24775）

**背景**：随着 agent 从"单次对话"演进为"长期运行的任务执行者"，记忆系统变得关键——如何跨会话保存信息、如何检索历史上下文、如何更新/遗忘旧知识？

**核心发现**：
- 评估了 12 个主流 agent 记忆架构（向量数据库、图结构、数据库式、混合式等）
- **没有单一架构能在所有任务类型上胜出**
- 不同记忆架构对不同工作负载有不同优势：
  - 向量检索：跨会话事实性（MemoryAgentBench）
  - 图结构：多跳关系推理（DB-Bench 中的关联查询）
  - 数据库式：程序性任务（LifelongAgentBench 的操作序列）
- 建议：先分析工作负载瓶颈（是跨会话事实检索？是关系推理？是程序执行？），再选记忆架构

**工程意义**：如果你正在为 agent 系统设计记忆层，这篇论文是目前最系统的对比资源。避免跟风选择"最新"的记忆方案，应该根据实际任务类型做决策。

## 5. LLM 可观测性进入成熟期

本周没有单一的"大新闻"，但几个信号合在一起说明 LLM 可观测性生态已经成熟：

- **Future AGI** 开源 SDK 新增对 13 种 guardrail 后端的支持，包括 Qwen3Guard 和 Granite Guardian 系列
- **Arize Phoenix** CLI 发布，终端直接查询 trace 数据，与 Cursor 和 Claude Code 的工作流集成更自然
- **Langfuse 被 Clickhouse 收购**（1 月事件，本周有更多用户反馈）：对数据主权有要求的团队需要关注架构变化，其他团队影响暂时有限
- **Confident AI** 发布了多轮对话 eval 的新 `TaskCompletionMetric`，专门评估 agent 是否"完成了用户真正的意图"而非只看单次响应质量

**信号解读**：当可观测性工具都在迭代"更细粒度的 agent 追踪"和"更自动化的评估"，说明开发者已经越过"让 LLM 能用"的阶段，进入"让 LLM 在生产环境中可靠运行并持续改进"的阶段。这是整个行业成熟的标志。
